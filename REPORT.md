# What and why:

Built here is an Aquarium hobbyist information site. The goal is to provide users, whether experienced fish keepers, or new to the hobby, with knowledge regarding the care necessary in maintaining a healthy, aquatic environment. User's may use the site to access information regarding tank stocking, compatible species, general maintenance, setup guides, aqua-scaping ideas, or just to ask the site's LLM for anything marine related.

Regarding the AI's behavior, the role of the AI is to provide accurate information specific to fish keeping. It should gather from the `knowledge_base`, using RAG to reference + gather the relevant info, and response in accordance with the user's input. Some difficulties, however, in getting the AI to behave go as follow:

    1. There's the general risk of bad advice, or misinformation being generated by the LLM.
    2. There are a huge variety of fish, plants, water types, locations, pricing, etc. etc., that the user could ask about, for the LLM to ascertain from - info mix ups could occur.
    3. In addition, due to various species, and their various varieties within (EX: Over 2,000 varieties of freshwater Minnow fish alone), the LLM can be too broad / robotic in its output, leading to users feeling gainless.
    4. The LLM has rules / keywords to follow by, to prevent users from prompting irrelevant topics, but this in turn could limit acceptable user input.

A lot of these errors occurred during the first iterations of the site.

# Code Walkthrough

A main component of the website is its `knowledge_base`; the information that the LLM would refer to to generate a useful response.

In the first iterations, the LLM would gather from `seed.py`, which contained keyword-specific prompts, with structured output (Line 182 to line 493), consisting of limited fish + plant + disease entries. The file `seed.py` was rejected, but remains as a schema for `ingest.py`, and as baseline data, to prevent a "no information found" response from the LLM, should `links.txt` be empty.

My original idea was for the LLM to extract from approved documents; to provide accurate information with little deception. So, I requested Kiro for a document / file insert file `ingest.py`, for external information to be included in the LLM's responses. Specifically Lines 243 to 283, which work as a URL extractor, where a document label, and link, is given to the LLM as context, if matching user's prompt. Thus with `links.txt`, the LLM opens URLs and reads the contents (using RAG), instead of the structured prompts from `seed.py`, as guidance to its responses.

# AI disclosure & safety

Kiro partook in the development of the backend, frontend, eval, and database of the site. The `knowledge_base` it generated however, I'd say was one of Kiro's inefficiencies. Upon first iteration, the AI implemented very basic, pre-generated responses to key-specific user inputs. As explained in the `Code Walkthrough` portion above, it didn't expand above the basic LLM use of "text in → structured response out".

    INPUT: "Guppy"

    OUTPUT: Basic response about water parameters needed, tank space, their behavior, feeding, other care needs.
        
While the response always come out the same (no hallucinations), what I want is more freedom in what the user can input, and a more varied, yet informational accurate response from the LLM call. However, admittedly, this structured approach is safer, and more controllable.

One safety risk, which could affect user's approval of the site, and the livelihood of the aquatic environment itself, is misinformation / hallucinations given by the site's AI model. Various testing of the LLM still resulted in either outdated, harmfully, or plain nonsense responses. The solution thus far is for the LLM to reference only approved docs `links.txt`, and for it to follow strict output rules, found in `prompt_factory.py`. This overall, constraints what can be an acceptable user input, narrowing to appropriate topic discussion from the LLM.
